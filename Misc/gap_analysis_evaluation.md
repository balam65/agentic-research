## Verified Gap Detail & Proposed Fixes

### 🔴 H1 — No Analysis → Implementation Bridge Ingestion Step

**Evidence of gap:**
- `08-research-onboarding-bridge.md` (analysis) explicitly outputs a unified `target-pipeline-config.yaml` and in §4 states: *"Push configurations to the Master Orchestrator's target registry."*
- However, `06-onboarding-sme.md` (implementation) has **zero reference** to consuming this YAML. Its 4 steps go directly from feasibility → evasion → knowledge schema → resource plan. There is no "Step 0: Ingest Analysis Bridge Configs."
- The bridge YAML fields (`extraction_engine`, `known_defenses`, `proxy_tier_required`, `alerting_thresholds`) are exactly the inputs the agents need — but there is no documented mechanism to load them.

**Risk if unfixed:** Human must manually copy YAML values into agent configs on every new target onboarding. This breaks the autonomous loop.

**How to fix:**
Add a new **Step 0** to `06-onboarding-sme.md`:

```markdown
### Step 0: Analysis Bridge Ingestion
**Objective:** Load the `target-pipeline-config.yaml` produced by the Research Analysis Framework 
into the live agent ecosystem before any onboarding steps begin.
**Prerequisites:** `08-research-onboarding-bridge.md` completed; `target-pipeline-config.yaml` available 
in the Master Orchestrator's target registry.
**Artifacts to produce:**
- `agentic-research/agents/onboarding/config_ingestion.py`
**Instruction:**
> Write `config_ingestion.py`. Read the `target-pipeline-config.yaml` for the active `job_id` 
> from the target registry. Map each config field to its destination: inject `extraction_engine` 
> and `known_defenses` into the scripting context, `proxy_tier_required` and `rate_limit_per_min` 
> into the production context, and `alerting_thresholds` into the sentinel baseline. 
> Log the ingestion event to `audit_logs`.
**Acceptance criteria:**
- All `target-pipeline-config.yaml` fields are distributed to their respective agent contexts.
- Ingestion event is logged with `job_id`, `target_domain`, and `timestamp`.
```

### 🔴 H2 — Scripting Agent Ignores Complexity Analysis Outputs

**Evidence of gap:**
- `03-extraction-complexity-analysis.md` (analysis) §4 explicitly states it produces metadata consumed by the **Scripting Agent**: tool stack (Playwright vs Axios), pagination logic, wait-states, fragility flags.
- `07-scripting-repository.md` (implementation) Step 2 (`script_generator.py`) says to *"take an HTML snippet and Target Field and use an LLM to generate XPath/CSS Selectors"* — it makes **no mention** of reading `03-extraction-complexity-analysis.md` or the `target-pipeline-config.yaml` for engine selection or wait-state rules.
- This means the Scripting Agent would independently re-decide the extraction engine rather than using the pre-computed architectural decision.

**Risk if unfixed:** Scripting Agent may select Axios (lightweight HTTP) on a Playwright-required target, causing guaranteed extraction failures.

**How to fix:**
Prepend a requirement to **Step 2 of `07-scripting-repository.md`**:

```markdown
**Instruction:**
> Before generating any selectors, read the active `target-pipeline-config.yaml` for this job. 
> Extract `extraction_engine`, `dynamic_wait_states`, and `known_defenses`. Use these as 
> non-negotiable architectural constraints for the generated script. Only after confirming the 
> engine selection, proceed to use an LLM to generate the optimal XPath/CSS selectors. Store 
> the engine selection and fragility flags alongside the selectors in the `scripts` table.
```

### 🔴 H3 — Sentinel Not Linked to Per-Target Readiness/Fragility Scores

**Evidence of gap:**
- `07-operational-readiness-assessment.md` (analysis) produces a per-target readiness scorecard including `Extraction Complexity` scores and fragility flags (A/B testing, ephemeral tokens, shadow DOM).
- `08-research-onboarding-bridge.md` (analysis) §2.3 explicitly generates a **Sentinel Spec Context** with `acceptable_extraction_null_rate` and `confidence_score_drop` — but this is documented in analysis only.
- `13-sentinel-monitoring.md` (implementation) `latency_tracker.py` only monitors generic job SLA elapsed time. It has **no reference** to per-target null rates, fragility levels, or confidence score drop thresholds.

**Risk if unfixed:** Sentinel treats a high-fragility (A/B-tested) target the same as a stable one. False negatives — the site breaks silently without alarm.

**How to fix:**
Add a new step to `13-sentinel-monitoring.md`:

```markdown
### Step 3: Per-Target Fragility Baseline Integration
**Objective:** Load target-specific alerting thresholds from the analysis bridge configs, 
rather than applying generic SLA floors.
**Artifacts to produce:**
- Update `latency_tracker.py` to read `alerting_thresholds` from `target-pipeline-config.yaml`
**Instruction:**
> Extend `latency_tracker.py`. For each active job, load the target's `alerting_thresholds` 
> block (from `target-pipeline-config.yaml`). Use `acceptable_extraction_null_rate` as the 
> null-rate ceiling and `confidence_score_drop` as the confidence floor, overriding 
> any generic system defaults for this target.
```

### 🟡 M1 — No Unified Target Registry Definition

**Evidence of gap:**
- `08-research-onboarding-bridge.md` §4 says to *"Push configurations to the Master Orchestrator's target registry"* — but **no document defines what this registry is**, where it lives, or its schema.
- `07-scripting-repository.md` defines `script_catalog_db.sql` and `06-onboarding-sme.md` defines `knowledge_repo_schema.yaml` — these are **two separate stores** with overlapping fields (domain, selectors, evasion rules) and no sync mechanism.

**How to fix:**
Add a short new implementation doc `00b-target-registry.md` or add a Step to `01-master-orchestration.md` that:
1. Defines a canonical `target_registry` table in Postgres (or as a YAML file for lightweight deployments).
2. Maps which fields are sourced from `script_catalog_db.sql` vs `knowledge_repo_schema.yaml`.
3. Designates the registry as the single source of truth that `config_ingestion.py` reads from.

### 🟡 M2 — Prototype Validation Loop Underspecified

**Evidence of gap:**
- `09-autonomous-research-execution-playbook.md` instructs agents to write a `prototype.py`.
- `07-scripting-repository.md` Step 2 mentions validating selectors via *"sample extraction"* before promoting to `ACTIVE` — but there is **no defined test harness, pass/fail criteria, or retry logic** for this promotion step.
- `08-extraction-orchestration.md` has no concept of a "staging/prototype" tier before ACTIVE.

**How to fix:**
Extend `07-scripting-repository.md` Step 2 acceptance criteria to define the promotion gate:
- Minimum N sample extractions required (e.g., 5).
- Required null rate below threshold (e.g., < 5%).
- Only then set `status = ACTIVE` in `script_catalog_db`.

### 🟡 M3 — Compliance Agent Not Wired to Compliance Delivery Analysis

**Evidence of gap:**
- `06-compliance-delivery-analysis.md` (analysis) §4 explicitly states it feeds the **Compliance Agent** with masking rules (regex), PII warnings, and rate limiting bounds.
- `02-compliance-security.md` (implementation) Step 2 generates `compliance_audit_rules.yaml` but instructs the agent to define its own regex patterns — it **never references** `06-compliance-delivery-analysis.md` as an input source.

**Risk if unfixed:** Masking rules and PII policies are reinvented per-target rather than inherited from analysis — inconsistent enforcement across targets.

**How to fix:**
Prepend to **Step 2 of `02-compliance-security.md`**:

```markdown
**Instruction:**
> Before defining regex patterns, read the `compliance.masking_enabled` and PII exposure flags 
> from the active `target-pipeline-config.yaml`. Pre-populate `compliance_audit_rules.yaml` 
> with any target-specific masking rules derived from the compliance analysis 
> (06-compliance-delivery-analysis.md). Then add the standard system-wide patterns 
> (API keys, AWS keys, credit cards, email addresses).
```

### 🟡 M4 — Resource Plan Ignores Cost Estimates from Readiness Assessment

**Evidence of gap:**
- `07-operational-readiness-assessment.md` (analysis) §2 produces a **daily cost estimate** broken down by compute, proxies, and LLM usage — with an explicit note: *"If cost exceeds client billing arrangements, block deployment until margin review."*
- `06-onboarding-sme.md` Step 4 creates `resource_deployment_plan.yaml` for compute/storage/proxy allocation — but **never references** the cost estimates from `07-operational-readiness-assessment.md`.

**Risk if unfixed:** High-cost targets (e.g., Playwright + residential proxies at scale) get deployed without a budget gate check.

**How to fix:**
Add to **Step 4 of `06-onboarding-sme.md`**:

```markdown
**Instruction:**
> Before finalising `resource_deployment_plan.yaml`, load the `Total Pipeline Weight` ($XXX/day) 
> from `07-operational-readiness-assessment.md`. Compare against the client's contracted billing 
> tier. If the estimated cost exceeds the client's margin floor, emit a `BUDGET_HOLD` signal to 
> the Master Agent and pause deployment pending human approval.
```
